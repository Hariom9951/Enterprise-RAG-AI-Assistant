## Overview

The Enterprise RAG AI Assistant allows enterprise users to:

1. **Upload documents** (PDF, DOCX, TXT) to a knowledge base.
2. **Ask questions** in natural language against that knowledge base.
3. **Receive grounded, cited answers** generated by a large language model.

**Phase 13 (current)** certifies the application as fully production-ready, featuring layout-aware text parsers, a recursive semantic chunker, hybrid search using Reciprocal Rank Fusion (RRF), streamed chat completions via Server-Sent Events (SSE) with citations, ReAct AI agents, and a Next.js playground user interface.

## Quick Start — Local Development

### 1. Backend Setup

```bash
# Navigate to the backend directory
cd backend

# Create a Python virtual environment
python -m venv .venv

# Activate the virtual environment
# On Windows:
.venv\Scripts\activate
# On macOS/Linux:
source .venv/bin/activate

# Install dependencies
pip install -r requirements.txt

# Create your local .env file from the template
cp .env.example .env
```

Configure your local `.env` settings, especially `DATABASE_URL` and `SECRET_KEY`.

---

### 2. Database Setup & Migrations

#### A. Running PostgreSQL
You can run PostgreSQL locally using standard installations, or spin up a PostgreSQL instance using Docker:
```bash
docker run --name pg-rag -e POSTGRES_USER=raguser -e POSTGRES_PASSWORD=ragpassword -e POSTGRES_DB=ragdb -p 5432:5432 -d postgres:16
```

#### B. Applying Migrations
Ensure your virtual environment is active in `/backend` and your `.env` contains the correct `DATABASE_URL`. Run the database migrations using Alembic:
```bash
# Apply migrations to update database schema to the latest version
alembic upgrade head
```

#### C. Startup the Server
```bash
# Start the development server (with hot-reload)
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

The API is now available at:
- **API Root**: http://localhost:8000/api/v1/
- **Health Check**: http://localhost:8000/api/v1/health
- **Swagger UI**: http://localhost:8000/docs
- **ReDoc**: http://localhost:8000/redoc

---

### 3. Frontend Setup

Open a **new terminal**:

```bash
# Navigate to the frontend directory
cd frontend

# Install Node.js dependencies
npm install

# Start the development server
npm run dev
```

The frontend is now available at: **http://localhost:3000**

- Sign in Page: **http://localhost:3000/login**
- Create Account Page: **http://localhost:3000/register**

## Running with Docker

Docker Compose runs the entire stack (backend + PostgreSQL 16 with pgvector + frontend) with a single command.

### Start All Services

```bash
# Build images and start all services in the foreground
docker-compose up --build

# Or run in detached (background) mode
docker-compose up --build -d
```

### Stop All Services

```bash
docker-compose down

# Also remove volumes (clears logs & local database data)
docker-compose down -v
```

## Authentication Flow

```
1. REGISTER ────────▶ POST /auth/register ──▶ Saves User (hashed password)
2. LOGIN ───────────▶ POST /auth/login ─────▶ Returns JWT Token pair (Access + Refresh)
3. ACCESS API ──────▶ Include Bearer <access_token> in "Authorization" Header
4. TOKEN EXPIRY ────▶ POST /auth/refresh ───▶ Rotates new Access + Refresh tokens
```

## API Reference

All API endpoints are prefixed with `/api/v1/`.

### Root & Health
- `GET /api/v1/` — Welcome endpoint.
- `GET /api/v1/health` — Returns status of API and database connection.

### Authentication
- `POST /api/v1/auth/register` — Create a new user account. Enforces email uniqueness and strong password policies (8+ chars, upper, digit, symbol).
- `POST /api/v1/auth/login` — Authenticate and obtain short-lived access token + long-lived refresh token.
- `POST /api/v1/auth/refresh` — Rotates and replaces expired tokens. Prevents reuse replay attacks (Token Rotation).

### Users
- `GET /api/v1/users/me` — Retrieve the profile of the currently logged-in active user.

### Knowledge Search & Retrieval
- `POST /api/v1/search` — Execute global hybrid search (combines semantic similarity with keyword match via RRF fusion). Supports threshold pruning, top_k, offset pagination, metadata/date/language filters.
- `POST /api/v1/search/batch` — Run hybrid searches for multiple query strings in a single batch.
- `POST /api/v1/documents/{document_id}/search` — Execute hybrid search scoped only to chunks of a specific document.
- `GET /api/v1/search/history` — Fetch search query audit trails log for active user.
- `GET /api/v1/search/statistics` — Fetch aggregated search stats and database latencies.

### Retrieval-Augmented Generation (RAG)
- `POST /api/v1/rag/query` — Run a RAG query to generate a grounded response answer with source citations.
- `POST /api/v1/rag/query/document/{document_id}` — Run a RAG query restricted only to context chunks from a single document.
- `GET /api/v1/rag/models` — Retrieve all supported active LLM models.
- `GET /api/v1/rag/statistics` — Fetch cumulative token counts and response times for the user.

## Environment Variables

Copy `backend/.env.example` to `backend/.env` and configure the values.

| Variable | Default | Description |
|----------|---------|-------------|
| `APP_NAME` | `Enterprise RAG AI Assistant` | Application display name |
| `APP_VERSION` | `0.2.0` | Semantic version string |
| `ENVIRONMENT` | `development` | `development` / `staging` / `production` |
| `DEBUG` | `false` | Enable debug mode (never `true` in prod) |
| `DATABASE_URL` | `postgresql+asyncpg://raguser:ragpassword@localhost:5432/ragdb` | SQLAlchemy Async database connection URI |
| `SECRET_KEY` | *(placeholder)* | JWT signature key (**MUST** be changed before production) |
| `ALGORITHM` | `HS256` | JWT signature algorithm |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | `30` | Access token TTL in minutes |
| `REFRESH_TOKEN_EXPIRE_DAYS` | `7` | Refresh token TTL in days |
| `LLM_PROVIDER` | `gemini` | Primary LLM Provider: `gemini`, `openai`, `ollama` |
| `GEMINI_API_KEY` | *(empty)* | Google Gemini services API key |
| `OPENAI_API_KEY` | *(empty)* | OpenAI services API key |
| `RAG_TOP_K` | `5` | Default number of source chunks to retrieve for prompt context |
| `RAG_MAX_CONTEXT_TOKENS` | `4000` | Token budget limit for prompt payload context |
| `RAG_TEMPERATURE` | `0.2` | Randomness temperature parameter for generation |
| `RAG_MAX_OUTPUT_TOKENS` | `1000` | Max tokens generated in completion |

## Running Tests

Tests run inside an isolated file-based SQLite database (`aiosqlite`) dynamically created per test module — no local Postgres setup is required for CI or local runs.

```bash
cd backend

# Activate virtual environment first
# source .venv/bin/activate (macOS/Linux)
# .venv\Scripts\activate.ps1 (PowerShell)

# Run the full test suite (110 tests)
pytest tests/ -v
```
