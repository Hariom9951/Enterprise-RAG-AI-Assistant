# Database Guidelines — Enterprise RAG AI Assistant

This document defines schema design patterns, migration workflows, index guidelines, and audit specifications for the PostgreSQL database.

---

## 1. UUID Policy

- **Primary Keys:** Every table in the system must use a UUID (Universally Unique Identifier) as its primary key. Auto-incrementing integers are prohibited for primary keys to prevent enumeration attacks and support offline ID generation.
- **SQLAlchemy Mapping:** Implement UUID columns using SQLAlchemy 2.0 `Uuid` type:
  ```python
  id: Mapped[uuid.UUID] = mapped_column(Uuid, primary_key=True, default=uuid.uuid4)
  ```
  On PostgreSQL, this generates native `UUID` datatypes. On SQLite, it automatically maps to a character fallback.

---

## 2. Naming Conventions

- **Tables:** lowercase snake_case plural nouns (e.g. `users`, `documents`, `chat_sessions`).
- **Columns:** lowercase snake_case singular names (e.g. `hashed_password`, `is_active`).
- **Foreign Keys:** Singular parent table name suffixed with `_id` (e.g. `user_id` inside `documents` table).
- **Constraints & Indices:** Alembic conventions automatically apply prefixes:
  - Primary Key: `pk_tablename`
  - Unique Constraint: `uq_tablename_columnname`
  - Index: `ix_tablename_columnname`

---

## 3. Relationships

- **Async Loading:** Always configure ORM relationships to load asynchronously. Avoid `lazy="joined"` or `lazy="select"` default loading which can cause synchronous blocking on async loop systems.
- **Explicit Cascades:** Declare delete cascades explicitly on relationships to prevent orphaned records (e.g. `cascade="all, delete-orphan"`).

---

## 4. Migrations (Alembic)

- **One Revision Per Change:** Group single schema actions (e.g. creating a table or adding a column index) into distinct, incremental migrations.
- **Autogenerate Inspection:** Always inspect the python code generated by `alembic revision --autogenerate` to verify data types, index targets, and rollback configurations before upgrading head.
- **Downgrade Safety:** Every migration must be reversible. Never write `pass` in a `downgrade()` implementation.

---

## 5. Indexing Strategy

- **Unique Fields:** Unique columns must contain a unique index (automatic for columns declared with `unique=True` or `PrimaryKeyConstraint`).
- **Foreign Keys:** Every foreign key column must be indexed. Databases do not automatically index foreign keys, leading to full-table scans during relational joins.
- **Search Optimization:** Fields queried frequently in `WHERE` clauses (like `email`, `created_at`) must contain explicit database indices.

---

## 6. Soft Delete Policy

- **No Hard Deletes:** Production transactional tables containing user documents, chats, or profile settings must use a soft delete mechanism to prevent accidental data loss.
- **Status Flag:** Implement soft deletes using a boolean status column (e.g. `is_deleted: Mapped[bool] = mapped_column(default=False)`) or a timestamp record (e.g. `deleted_at: Mapped[datetime | None]`).
- **Global Filters:** When querying tables, configure SQLAlchemy select statements to filter out soft-deleted records by default.

---

## 7. Audit Fields

Every table must inherit the audit timestamp properties from `TimestampMixin`:
- `created_at` — DateTime column capturing insertion time. Declared with `server_default=func.now()` to ensure the database sets the timezone-aware timestamp.
- `updated_at` — DateTime column updating on every save. Declared with `server_default=func.now()`, `onupdate=func.now()` (or mapped trigger handles in PostgreSQL) to update timestamps automatically on database mutations.
