# Contributing Guidelines — Enterprise RAG AI Assistant

Thank you for contributing! To maintain production-grade software standards, all contributions must follow this contributing guide.

---

## 1. Branch Naming Strategy

We use a structured branch naming convention prefixing branches with their intent:
- `feature/` — New features or endpoints (e.g., `feature/auth-jwks`).
- `bugfix/` — Fixing a bug or resolving an issue (e.g., `bugfix/token-refresh-leak`).
- `hotfix/` — Critical production patches (e.g., `hotfix/secret-key-exposure`).
- `docs/` — Documentation updates (e.g., `docs/api-guidelines`).
- `refactor/` — Code cleanup or formatting adjustments without functional changes (e.g., `refactor/db-session-reorganization`).

---

## 2. Commit Message Convention

We strictly follow the **Conventional Commits** specification. Commits should be structured as follows:

```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

### Types
- `feat`: A new feature (corresponds to a minor release).
- `fix`: A bug fix (corresponds to a patch release).
- `docs`: Documentation-only changes.
- `style`: Changes that do not affect the meaning of the code (white-space, formatting, missing semi-colons, etc.).
- `refactor`: A code change that neither fixes a bug nor adds a feature.
- `test`: Adding missing tests or correcting existing tests.
- `chore`: Changes to the build process or auxiliary tools/libraries (e.g., updating dependencies).

### Examples
- `feat(auth): add role-based access control checking middlewares`
- `fix(db): resolve database connection pool exhaustion on close`
- `docs(api): document query parameters for registration endpoints`

---

## 3. Pull Request Checklist

Before submitting a Pull Request, ensure that:
1. [ ] **Lint checks pass:** `ruff check app/ tests/` (Python) and `npm run lint` (TypeScript) return no errors.
2. [ ] **Type assertions pass:** `mypy app/ --ignore-missing-imports` resolves with 0 errors.
3. [ ] **Test coverage is maintained:** All 32+ unit/integration tests pass with `pytest`.
4. [ ] **Migrations are clean:** New database models contain corresponding Alembic migration revisions.
5. [ ] **Documentation is updated:** Appropriate files inside `docs/` have been updated.
6. [ ] **Branch is rebased:** Rebased onto the latest `main` branch to avoid merge conflicts.

---

## 4. Code Review Checklist

Reviewers must evaluate PRs based on:
- **Clean Architecture:** Is there a separation of concerns? (e.g., endpoints should never talk directly to SQLAlchemy database drivers — they must delegate to services).
- **Type Safety:** Are Python type hints and TypeScript interfaces fully implemented? No use of `Any` without explicit casts or annotations.
- **SQL Injection:** Are raw queries avoided? (Prefer SQLAlchemy expression language).
- **Error Handling:** Are exceptions caught and converted to RFC-compliant error responses? (No raw server tracebacks returned to users).
- **Resource Cleanup:** Are database sessions, file handles, and networking streams closed gracefully via context managers or lifespan blocks?

---

## 5. Testing Requirements

- **Unit Tests:** Must cover all services and utility modules.
- **Integration Tests:** Must cover API endpoints including authentication, parameter validation, role checking, and database queries.
- **Mocking:** Use in-memory databases (SQLite via `aiosqlite`) to test endpoints without requiring external services.
- **Coverage:** Aim for at least **90% code coverage** for all new code.

---

## 6. Folder Organization Rules

Keep directories clean and hierarchical:
- `backend/app/api/v1/endpoints/` — Endpoint controllers only (parse inputs, call services, return schemas).
- `backend/app/services/` — Business logic (computations, DB interactions, external APIs).
- `backend/app/models/` — Database ORM mapping models.
- `backend/app/schemas/` — Request/response validation schemas (Pydantic models).
- `frontend/src/components/` — Reusable, atomic UI components.
- `frontend/src/app/` — Next.js routing pages.

---

## 7. API Naming Conventions

- **RESTful Endpoints:** Plural nouns for collections (e.g., `/api/v1/users`), singular nouns for specific elements (e.g., `/api/v1/users/{user_id}`).
- **HTTP Methods:**
  - `GET` — Retrieve resource (idempotent).
  - `POST` — Create a new resource.
  - `PUT` — Replace an entire resource.
  - `PATCH` — Partially update a resource.
  - `DELETE` — Remove a resource.
- **URL Paths:** Use lowercase kebab-case (e.g., `/api/v1/user-profiles`).

---

## 8. Database Migration Policy

- **No Manual Schema Modifications:** All schema changes must go through Alembic.
- **Autogenerated Scripts Review:** Autogenerated migration scripts must be inspected for correct syntax, column comments, indices, and drop safety.
- **One Migration Per Change:** Do not group unrelated table modifications into a single revision.
- **Reversible:** Every migration must implement both an `upgrade()` and a `downgrade()` function.
